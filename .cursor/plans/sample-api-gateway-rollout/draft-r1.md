# API 网关限流计划

## 任务理解

需要为新的 API 网关限流能力给出一份首期开发计划，覆盖功能边界、实施路径和验证方案。

## 目标

- 为外部 API 提供按租户和按路由的限流能力
- 在不改动业务服务的前提下先落到网关层
- 支持灰度发布和回滚

## 范围

- 首期只支持固定窗口限流
- 限流决策发生在网关入口
- 首期只覆盖公开 API，不覆盖内部服务间调用

## 假设

- 平台已经提供统一的分布式限流服务，可直接复用
- 每个请求都能稳定拿到租户 ID 和路由 ID

## 风险

- 限流键设计不当会导致热点
- 错误返回策略不清晰会影响客户端重试行为

## 建议方案

在网关请求入口增加限流拦截器，按 `tenantId + routeId` 组合键查询统一限流服务。若超过阈值，则直接返回 `429`，并附带标准化错误码。配置中心维护各路由阈值，首期通过灰度开关控制是否对某个租户生效。

## 验证方式

- 通过压测验证超过阈值时返回 `429`
- 通过灰度环境验证不同租户使用不同阈值

## 开放问题

- 是否需要首期就支持返回 `Retry-After`

## 执行步骤
1. 定义路由级限流配置模型和灰度开关。
2. 在网关入口增加限流拦截器并接入统一限流服务。
3. 为公开 API 路由接入配置并在灰度环境压测。
