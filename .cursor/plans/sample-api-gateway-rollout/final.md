# API 网关限流计划

## 任务理解

需要为新的 API 网关限流能力给出一份首期开发计划，覆盖功能边界、实施路径和验证方案。

## 目标

- 为外部 API 提供按租户和按路由的限流能力
- 在不改动业务服务的前提下先落到网关层
- 支持灰度发布和回滚

## 范围

- 首期只支持固定窗口限流
- 限流决策发生在网关入口
- 首期只覆盖公开 API，不覆盖内部服务间调用

## 假设

- 每个请求都能稳定拿到租户 ID 和路由 ID
- 网关所在环境可以访问现有 Redis 集群

## 风险

- 限流键设计不当会导致热点
- Redis 故障时需要确定降级策略，避免把流量全部错误拒绝

## 建议方案

在网关请求入口增加限流拦截器，按 `tenantId + routeId` 组合键在 Redis 中执行固定窗口计数。若超过阈值，则直接返回 `429`，并附带标准化错误码。配置中心维护各路由阈值，首期通过灰度开关控制是否对某个租户生效。若 Redis 不可用，则按“记录告警并放行”降级，避免引入大面积误拒绝。

## 验证方式

- 功能正确性：超过阈值时 `429` 命中率达到 100%
- 性能影响：灰度路由的 P95 延迟增量不超过 5 ms
- 风险控制：误拦截率低于 0.1%
- 灰度验收：至少覆盖 3 个租户和 5 条公开 API 路由

## 开放问题

- 是否需要首期就支持返回 `Retry-After`

## 执行步骤
1. 定义路由级限流配置模型、灰度开关和 Redis key 规则。
2. 在网关入口增加限流拦截器，并实现 Redis 固定窗口计数逻辑。
3. 增加 Redis 不可用时的告警与放行降级策略。
4. 为公开 API 路由接入配置并在灰度环境完成压测与验收。
